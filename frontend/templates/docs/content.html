Introduction

Overview

Sentinel is a real-time multi-domain surveillance dashboard for tracking activity across Air, Space, Sea, Land, and SDR domains. It combines a FastAPI backend with a MapLibre GL browser client to display live data on interactive maps.

The application is designed for offline-first operation: every domain supports configurable online and offline data sources, with automatic fallback when connectivity changes. A global connectivity mode and per-domain source overrides allow fine-grained control.

All HTTP traffic enters via nginx (port 80/443 in Docker) which proxies to uvicorn on port 8000. Static assets are served directly by nginx from the /frontend/assets and /frontend/components directories. Page requests hit FastAPI which renders Jinja2 templates. API requests are handled by domain routers.

Connectivity Model

High-level — source resolution logic
flowchart TD A[Request for domain data] --> B{Domain sourceOverride?} B -->|online| ON[Use online URL] B -->|offline| OF[Use offline URL] B -->|auto| C{Global connectivityMode?} C -->|online| ON C -->|offline| OF ON --> D{Upstream reachable?} D -->|yes| E[Return fresh data] D -->|stale window ok| F[Return stale cache] D -->|stale expired| G[Return error] OF --> H[Use local network URL]

Each domain has two data source slots — online and offline. A global connectivityMode setting (online/offline) selects which slot is active. Individual domains can override this with a per-domain sourceOverride (auto / online / offline). resolve_domain_urls() in utils.py encapsulates this logic and returns the effective (primary_url, fallback_url) pair for any request.

Domains

Domains

AIR

High-level — AIR domain data flow
flowchart LR FE[Browser MapLibre] -->|GET adsb point| BE[FastAPI air router] BE --> CK{Cache fresh?} CK -->|HIT| CR[Return cached JSON] CK -->|STALE| ST[Return stale JSON] CK -->|MISS| UP[airplanes.live API] UP --> WR[Write to adsb_cache] WR --> RS[Return fresh JSON] FE -->|squawk events| MSG[air_messages] FE -->|add or remove| TRK[air_tracking]
Low-level — ADS-B cache TTL state machine
stateDiagram-v2 [*] --> MISS : No entry in adsb_cache MISS --> FRESH : Upstream fetched and written FRESH --> STALE : age exceeds adsb_ttl_ms 5s STALE --> FRESH : Upstream fetched and refreshed STALE --> EXPIRED : age exceeds adsb_stale_ms 30s EXPIRED --> MISS : Entry removed or ignored

Real-time ADS-B aircraft tracking powered by the airplanes.live API. The backend proxies requests to the upstream and caches responses in SQLite to reduce upstream calls (5 s TTL, 30 s stale window).

Aircraft are rendered as icons on a MapLibre map. Clicking an aircraft opens a detail panel with callsign, altitude, speed, and heading. Aircraft can be added to the tracking list which persists across sessions.

Notification messages (emergencies, special squawks) are stored in air_messages and displayed in the notification panel.

SPACE

High-level — SPACE domain data flow
flowchart LR FE[Browser MapLibre] -->|GET space iss| Router[FastAPI space router] Router --> SAT[satellite.py SGP4] SAT --> TLEC[tle_cache SQLite] TLEC -->|stale or empty| CTK[Celestrak API] CTK --> TLEC SAT --> POS[ISS position and ground track] Router -->|GET space daynight| DN[daynight.py GeoJSON] FE -->|upload or fetch URL| TM[TLE management API] TM --> TLEC
Low-level — TLE ingest and propagation lifecycle
flowchart TD A[POST tle/fetch or tle/manual] --> B[Parse TLE text and validate lines] B --> C{NORAD ID already in DB?} C -->|yes| D[Update record and reset fetched_at_ms] C -->|no| E[Insert new record] D --> F[satellite_catalogue upsert] E --> F G[GET space/iss] --> H{tle_cache for ISS fresh?} H -->|yes| I[Propagate with sgp4] H -->|no| J[Fetch from Celestrak and update tle_cache] J --> I I --> K[Return position ground track and footprint GeoJSON]

Satellite tracking using SGP4 propagation from Two-Line Element (TLE) data. The ISS is tracked by default. TLE data is fetched from Celestrak and cached for 6 hours (stale window: 12 hours).

The day/night terminator is rendered as a GeoJSON overlay. TLE data can be managed via the Settings panel: fetch from a URL, upload a .txt file, assign categories to satellites, and clear the database.

Satellite categories: space_station, amateur, weather, military, navigation, science, cubesat, active, unknown.

The Satellite Filter panel (opened via the filter button in the space side menu) provides real-time search across all satellites stored in the TLE database. Typing in the panel searches by satellite name, NORAD ID, or category — including human-readable aliases (e.g. gps matches navigation, ham matches amateur). Selecting a result switches the active satellite on the map. Results are capped at 50 for performance; the panel pre-loads the satellite list on init and refreshes it on each open. Keyboard navigation (↑ ↓ Enter Escape) is supported.

Config File (default_config.json)

backend/default_config.json is the seed file loaded on first startup to populate the database with default settings. You can download the current live settings at any time via Settings › Config › Download, modify it, then re-upload via Settings › Config › Upload to replace all settings at once.

The file is a JSON object whose top-level keys are namespaces — one per domain plus a global app section. Each namespace contains key/value pairs that map directly to the settings stored in the database.

app — Global application settings
KeyTypeDescription
connectivityMode "online" | "offline" Global data-source selector. online uses upstream internet feeds; offline routes all domains to their configured local/offline source URL. Individual domains can override this with sourceOverride.
connectivityProbeUrl URL string A lightweight URL polled every 2 seconds to detect internet connectivity. A failed request turns the connectivity indicator red. Defaults to the OpenStreetMap tile favicon — any reliably reachable URL works.
air — ADS-B aircraft tracking
KeyTypeDescription
enabled boolean Whether the AIR domain is active. When false the domain is skipped entirely — no data is fetched and the map layer is not rendered.
sourceOverride "auto" | "online" | "offline" Per-domain source selector. auto follows the global connectivityMode. online or offline force this domain to use a specific source regardless of the app-level mode.
onlineUrl URL string Base URL for the live ADS-B upstream API. The backend appends /point/{lat}/{lon}/{radius} to this URL when fetching aircraft. Defaults to https://api.airplanes.live/v2.
space — Satellite tracking
KeyTypeDescription
enabled boolean Whether the SPACE domain is active.
sourceOverride "auto" | "online" | "offline" Per-domain source selector — same behaviour as the AIR domain override.
onlineUrl URL string Default Celestrak TLE feed URL used when fetching satellite data. Acts as a fallback; per-category URLs in onlineUrls take precedence when a specific category is requested.
onlineUrls object Per-category Celestrak TLE feed URLs. Each key is a satellite category name and the value is the full Celestrak URL for that group. Supported categories: space_station, amateur, weather, military, navigation, science, cubesat, active, unknown.
sea — Maritime tracking (stub)
KeyTypeDescription
enabled boolean Whether the SEA domain is active. Infrastructure is in place but data source integration is not yet implemented.
sourceOverride "auto" | "online" | "offline" Per-domain source selector — will be respected when SEA functionality is implemented.
land — Land tracking (stub)
KeyTypeDescription
enabled boolean Whether the LAND domain is active. Infrastructure is in place but data source integration is not yet implemented.
sourceOverride "auto" | "online" | "offline" Per-domain source selector — will be respected when LAND functionality is implemented.
sdr — Software-defined radio (stub)
KeyTypeDescription
enabled boolean Whether the SDR domain is active.
onlineUrl URL string Base URL for an online SDR data feed.
offlineSource object — { "url": string } Configuration for a local/offline SDR source. The nested url field should point to a self-hosted SDR server (e.g. http://192.168.1.x:port).

Infrastructure

Deployment

Deployment Architecture

High-level — Docker deployment topology
flowchart TD Browser([Browser]) -->|HTTP port 80| nginx subgraph Compose ["docker-compose"] nginx[nginx container] -->|proxy_pass port 8000| FastAPI[FastAPI uvicorn] nginx -->|serve direct| Static[frontend/assets] FastAPI --- Volume[(sentinel.db volume)] end

Docker (recommended)

The project ships with a docker-compose.yml that runs nginx as a reverse proxy in front of the FastAPI container. 

docker compose up --build

The app will be available at http://localhost (port 80). nginx serves static files directly and proxies all other requests to uvicorn on port 8000.

Local Development

Run the FastAPI backend with hot-reload: 

cd /path/to/Sentinel
python -m venv backend/.venv
source backend/.venv/bin/activate
pip install -e "backend[dev]"
uvicorn backend.main:app --reload --port 8000

The app will be available at http://localhost:8000. FastAPI's interactive API docs (Swagger UI) are at http://localhost:8000/docs.

Frontend Build

TypeScript source files in frontend/components/ are compiled to plain JavaScript. The compiled .js files are committed alongside the .ts sources. 

# One-time build
npm run build

# Watch mode (recompiles on save)
npm run watch

# Type-check only
npm run typecheck

nginx Config

nginx.conf proxies all requests to uvicorn. Static files under /assets/ are served directly with long cache headers. The proxy passes standard forwarding headers (X-Real-IP, X-Forwarded-For).

Environment Variables

All backend/config.py settings can be overridden at runtime via environment variables (uppercase, same name) or a .env file. For Docker deployments, set them in docker-compose.yml under the environment: key. 

# Example .env
ADSB_UPSTREAM_BASE=https://my-custom-feed.example.com/v2
DB_PATH=/data/sentinel.db
TLE_TTL_MS=3600000
