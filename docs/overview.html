Purpose

SENTINEL is a military and civil surveillance map application that renders live air traffic, airspace zones, airports, and RAF bases on an interactive map. It is designed for situational awareness over the UK and surrounding airspace.

The application is built as a single-page app with a modular vanilla JavaScript frontend backed by a FastAPI Python REST API. Three further domains — space, sea, and land — are architecturally scaffolded but not yet implemented.

High-Level Architecture

Key Architectural Decisions

No build step

JavaScript files are loaded directly via <script> tags in a defined order. TypeScript is compiled with tsc but not bundled. This keeps development fast and avoids toolchain complexity.

IIFE / window globals pattern

Shared components (_Notifications, _Tracking, _FilterPanel, MapComponent) are exposed on window via IIFEs, giving global access without ES module imports.

Backend as proxy + cache

The FastAPI backend does not own the source data — it proxies external APIs (airplanes.live, Nominatim) and caches responses in SQLite to reduce upstream load and provide stale fallback when APIs are unavailable.

Data Flow

ADS-B Aircraft Tracking

After receiving aircraft data, AdsbLiveControl interpolates positions every 100ms using dead-reckoning, renders GeoJSON symbol layers, checks for emergency squawk codes, and updates the tracking panel.

User Location Flow

Online / Offline Mode

SENTINEL detects internet connectivity by polling the OSM favicon every 2 seconds. On connectivity change, the map style is switched:

ModeMap StyleSourceZoom Default
Onlinefiord-online.jsonOpenStreetMap tile server6
Offlinefiord.jsonLocal PMTiles bundle5

When the style switches, overlay-reinit.js re-initializes all overlays by calling each control's initLayers() method.

Offline bounds: PMTiles data covers [[-20, 44], [32, 67]] — UK, Ireland, and surrounding European waters. Panning outside this box while offline shows blank tiles.

Deployment

Two Docker containers are defined in docker-compose.yml:

ContainerImagePortPurpose
sentinel-apiBuilt from backend/Dockerfile8000FastAPI + Uvicorn; serves /api/* endpoints and static files
sentinelBuilt from root Dockerfile (nginx:alpine)8080Nginx serving frontend static files

docker-compose.yml

version: '3.8'
services:
  backend:
    build:
      context: .
      dockerfile: backend/Dockerfile
    container_name: sentinel-api
    ports:
      - "8000:8000"
    volumes:
      - sentinel_db:/app/data       # SQLite persisted outside container
    environment:
      - DB_PATH=/app/data/sentinel.db
    restart: unless-stopped

  map-server:
    build: .                        # uses root Dockerfile (nginx)
    container_name: sentinel
    ports:
      - "8080:80"
    depends_on:
      - backend
    restart: unless-stopped

volumes:
  sentinel_db:                      # named volume — survives container rebuilds

Root Dockerfile (nginx frontend)

FROM nginx:alpine
COPY . /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf

Copies all project files into nginx's web root and applies a custom nginx.conf. The config is expected to proxy /api requests to the backend container at sentinel-api:8000.

Quick Start

# Build and start both containers
docker compose up --build

# Backend API:   http://localhost:8000
# Frontend:      http://localhost:8080
# Swagger UI:    http://localhost:8000/docs

# Stop
docker compose down

# Stop and remove database volume (destructive)
docker compose down -v
The SQLite database is persisted in a named Docker volume sentinel_db mounted at /app/data/sentinel.db inside sentinel-api. Rebuilding or removing the container does not delete the database unless you also remove the volume (docker compose down -v).
