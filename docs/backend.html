Backend Overview

The backend is a FastAPI application running on Python 3.12 with an async SQLAlchemy ORM backed by SQLite. It acts primarily as a proxy and cache — it does not own source data but fetches from upstream APIs and caches results to reduce load and provide degraded service when APIs are unavailable.

main.py — Application Entry Point

Creates the FastAPI application, registers all domain routers, mounts static file serving, and runs the startup lifecycle.

async lifespan(app: FastAPI) async context manager
Runs create_tables() on startup. FastAPI calls this before accepting requests. Tear-down logic (if any) goes after the yield.
Calls: database.create_tables()
GET /health → {"status": "ok", "timestamp": int}
Health check endpoint. Returns current Unix timestamp in seconds. Used by orchestration (Docker healthcheck, load balancer probes).

cache.py — Cache Helpers

now_ms() → int
Returns current Unix time in milliseconds. Used as the base for all TTL calculations.
is_fresh(expires_at: int) → bool
Returns True if now_ms() < expires_at. Used to check if cached data is still within its primary TTL.
is_within_stale(fetched_at: int, stale_ms: int) → bool
Returns True if data is old but within its stale window. Used as a fallback when upstream API is unavailable.

ADS-B Cache Strategy

Write-through with stale fallback:
1. Check SQLite for a row matching cache key.
2. If is_fresh(expires_at) → return cached payload with X-Cache: HIT.
3. If expired → call adsb_service.fetch_aircraft_from_upstream().
4. On upstream success → upsert to SQLite, return with X-Cache: MISS.
5. On upstream failure → if is_within_stale(fetched_at, adsb_stale_ms) → return stale with X-Cache: STALE.
6. Otherwise → raise HTTP 503.

routers/settings.py — User Settings Router

All endpoints are prefixed /api/settings. Persists and retrieves user preferences in the user_settings SQLite table. Full endpoint details are in the API Reference.

Endpoints

MethodPathDescription
GET/api/settingsReturns all settings grouped by namespace as { namespace: { key: value } }
GET/api/settings/{namespace}Returns all settings for one namespace as { key: value }
PUT/api/settings/{namespace}/{key}Upserts a single setting. Body: { "value": any }. Returns { "status": "ok" }.

Value Serialisation

All values are stored as JSON strings in the value column and deserialised before returning. This means the API transparently handles booleans (true/false), strings ("dark"), and objects ({"airports": true}) without the caller needing to manage serialisation.

Fire-and-forget design: The frontend window._SettingsAPI client catches all errors from these endpoints and logs them to the console without propagating them. A settings persistence failure never interrupts the UI.

services/adsb.py — ADS-B Client

async fetch_aircraft_from_upstream(lat: float, lon: float, radius: int) → dict
HTTP GET to {adsb_upstream_base}/point/{lat}/{lon}/{radius}. Sets User-Agent: SENTINEL/1.0. Raises httpx.HTTPError on failure. Returns raw JSON dict.
Uses: httpx.AsyncClient, settings.adsb_upstream_base

fetch_aircraft is an alias for backward compatibility.

services/geocode.py — Nominatim Client

async reverse_geocode_coordinates(lat: float, lon: float) → dict
HTTP GET to {nominatim_base}/reverse?format=json&lat={lat}&lon={lon}. Sets User-Agent: SENTINEL/1.0 and Accept-Language: en. Returns raw Nominatim JSON. The address.country field is used to update the footer location label in the frontend.
Uses: httpx.AsyncClient, settings.nominatim_base

reverse_geocode is an alias for backward compatibility.

backend/Dockerfile — Container Image

Builds the sentinel-api container. Based on python:3.12-slim with uv for fast, reproducible dependency installation.

FROM python:3.12-slim

WORKDIR /app

# Install uv (from Astral's official image)
COPY --from=ghcr.io/astral-sh/uv:latest /uv /uvx /bin/

# Install Python dependencies from lockfile (no network drift)
COPY backend/pyproject.toml backend/uv.lock ./
RUN uv sync --frozen --no-dev

# Copy backend source
COPY backend/ ./backend/

# Copy static files so FastAPI can serve the frontend in dev/single-container mode
COPY index.html main.js style.css ./
COPY assets/ ./assets/

# Create persistent data directory (SQLite volume mount point)
RUN mkdir -p /app/data

CMD ["uv", "run", "uvicorn", "backend.main:app", "--host", "0.0.0.0", "--port", "8000"]

Key Points

DetailNotes
Base imagepython:3.12-slim — minimal Debian-based image
Dependency manageruv installed from ghcr.io/astral-sh/uv:latest. uv sync --frozen --no-dev installs exactly what's in uv.lock, no dev dependencies.
Static file servingFrontend files (index.html, style.css, assets/) are copied in so the backend can serve the whole app from a single container in dev mode.
Data directory/app/data/ is created for the SQLite file. In docker-compose, the named volume sentinel_db is mounted here so the database persists across container rebuilds.
Start commandUvicorn on 0.0.0.0:8000. The port is exposed by docker-compose as 8000:8000.
In production, the sentinel nginx container (built from the root Dockerfile) reverse-proxies /api requests to the backend. The backend's static file serving is mainly useful for local development without running the nginx container.

Database Storage

SENTINEL uses a single SQLite file (sentinel.db) for all persistence — ADS-B cache, geocode cache, notification messages, and aircraft tracking state.

File Location

EnvironmentPath
Local dev (default)backend/sentinel.db (relative to project root)
Docker container/app/data/sentinel.db (inside the sentinel-api container)
Docker host (volume)Named volume sentinel_db → inspect with docker volume inspect sentinel_db
The database file is created automatically by create_tables() on first startup. If the file does not exist, SQLAlchemy will create it. No migration step is needed for a fresh install.

Table Overview

TableRowsPurposeGrows?
adsb_cache1 per query regionStores latest ADS-B JSON per lat/lon/radius key. Upserted on every cache miss — stays small.Bounded (upsert)
geocode_cache1 per ~1km cell visitedNominatim reverse-geocode results. Accumulates as new locations are visited.Slow growth
air_message1 per notification ever shownNotification history including dismissed ones (soft-delete). Can grow unbounded in long-running deployments.Grows over time
air_tracking1 per tracked aircraftCurrently tracked aircraft. Small — deleted when aircraft is untracked.Small / bounded
user_settings1 per namespace/key pairUser preferences persisted across sessions (theme, overlay states, etc.). Upserted on every change — never grows beyond the number of distinct settings.Bounded (upsert)

Connecting to the Database

# From project root

SELECT cache_key, ac_count, fetched_at FROM adsb_cache;
.quit

Local Development — DB Browser for SQLite (GUI)

DB Browser for SQLite is a free cross-platform GUI. Open backend/sentinel.db directly from the file menu. The Browse Data tab lets you filter, sort, and export any table.

Docker — Accessing the Database

The database lives inside the sentinel-api container. Use one of these approaches:

# Option 1: exec into the running container and use sqlite3
docker exec -it sentinel-api sqlite3 /app/data/sentinel.db ".tables"

# Option 2: copy the database file out to your host
docker cp sentinel-api:/app/data/sentinel.db ./sentinel_local_copy.db
sqlite3 sentinel_local_copy.db

# Option 3: find the Docker volume path on macOS/Linux
docker volume inspect sentinel_db
# Look for "Mountpoint" — the absolute path on the host filesystem
# (may require sudo to access depending on your Docker setup)

Python — Direct SQLAlchemy Access

For scripting or debugging, connect using SQLAlchemy or aiosqlite directly:

import sqlite3
conn = sqlite3.connect("backend/sentinel.db")
cur = conn.cursor()

# List non-dismissed notifications
cur.execute("SELECT msg_id, type, title, ts FROM air_message WHERE dismissed=0")
for row in cur.fetchall():
    print(row)

conn.close()

FastAPI Interactive Docs

The running backend exposes Swagger UI at http://localhost:8000/docs. You can query all API endpoints (including tracking and messages) directly from the browser — changes are immediately reflected in the database.

Schema Reference

Full CREATE TABLE statements as generated by SQLAlchemy. Column types map to SQLite affinity rules.

adsb_cache

CREATE TABLE adsb_cache (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    cache_key  VARCHAR NOT NULL UNIQUE,   -- e.g. "54.1453_-4.4815_250"
    lat        FLOAT NOT NULL,
    lon        FLOAT NOT NULL,
    radius_nm  INTEGER DEFAULT 250,
    payload    TEXT NOT NULL,             -- raw JSON string from airplanes.live
    ac_count   INTEGER NOT NULL,
    fetched_at BIGINT NOT NULL,           -- Unix milliseconds
    expires_at BIGINT NOT NULL            -- fetched_at + adsb_ttl_ms
);

geocode_cache

CREATE TABLE geocode_cache (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    cache_key  VARCHAR NOT NULL UNIQUE,   -- e.g. "54.14_-4.48"
    lat        FLOAT NOT NULL,
    lon        FLOAT NOT NULL,
    raw        TEXT NOT NULL,             -- full Nominatim JSON
    fetched_at BIGINT NOT NULL,
    expires_at BIGINT NOT NULL
);

air_message

CREATE TABLE air_message (
    id        INTEGER PRIMARY KEY AUTOINCREMENT,
    msg_id    VARCHAR NOT NULL UNIQUE,    -- client-generated idempotency key
    type      VARCHAR NOT NULL,           -- "emergency" | "flight" | "system" | "squawk-clr"
    title     VARCHAR NOT NULL,
    detail    VARCHAR DEFAULT '',
    ts        BIGINT NOT NULL,            -- Unix milliseconds (event time)
    dismissed BOOLEAN DEFAULT 0           -- soft-delete flag
);

air_tracking

CREATE TABLE air_tracking (
    id       INTEGER PRIMARY KEY AUTOINCREMENT,
    hex      VARCHAR NOT NULL UNIQUE,     -- ICAO 24-bit hex e.g. "400f86"
    callsign VARCHAR DEFAULT '',
    follow   BOOLEAN DEFAULT 0,           -- camera-follow mode
    added_at BIGINT NOT NULL              -- Unix milliseconds
);

user_settings

CREATE TABLE user_settings (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    namespace  TEXT NOT NULL,       -- 'app' | 'air' | 'space' | 'sea' | 'land' | 'sdr'
    key        TEXT NOT NULL,       -- e.g. 'theme', 'overlayStates'
    value      TEXT NOT NULL,       -- JSON-serialised value
    updated_at INTEGER NOT NULL,    -- Unix milliseconds
    UNIQUE (namespace, key)         -- composite unique constraint; PUT is an upsert
);

Useful Diagnostic Queries

-- See what's currently in the ADS-B cache
SELECT cache_key, ac_count,
       datetime(fetched_at/1000, 'unixepoch') AS fetched,
       datetime(expires_at/1000, 'unixepoch') AS expires
FROM adsb_cache;

-- Count notifications by type
SELECT type, COUNT(*) as count, SUM(dismissed) as dismissed
FROM air_message GROUP BY type;

-- List all tracked aircraft
SELECT hex, callsign, follow,
       datetime(added_at/1000, 'unixepoch') AS added
FROM air_tracking ORDER BY added_at DESC;

-- Check geocode cache coverage
SELECT COUNT(*) as cached_cells FROM geocode_cache;
SELECT cache_key FROM geocode_cache ORDER BY fetched_at DESC LIMIT 10;

-- List all persisted user settings
SELECT namespace, key, value,
       datetime(updated_at/1000, 'unixepoch') AS last_updated
FROM user_settings ORDER BY namespace, key;

-- Get all settings for the 'app' namespace
SELECT key, value FROM user_settings WHERE namespace = 'app';

Cache Strategy Detail

ResourceFresh TTLStale WindowCache Key
ADS-B5 seconds30 seconds"{lat}_{lon}_{radius}"
Geocode10 minutes1 hour"{lat_2dp}_{lon_2dp}" (1km grid)
The geocode cache key is rounded to 2 decimal places (~1km resolution). This means nearby queries (e.g. 54.1453 and 54.1478) resolve to the same cache entry, reducing Nominatim API calls significantly.
